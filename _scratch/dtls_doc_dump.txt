### [Heading1] Purpose
The purpose of this document is to outline the necessary steps/guidelines for creating security roles with the correct authorizations relevant to DTLS. Also, this document will cover creating Fiori groups and catalogs for the purposes of assigning them to DTLS roles. 

### [Heading1] Fiori Configuration
Fiori configuration consists of creating the necessary Catalogs, Groups, Spaces, and Pages required for providing the end users with the ability to access the Applications (Tiles) in the Fiori Launchpad.
It’s important to note that, depending on your S/4HANA version, there may be a difference in how the Tiles are organized and presented to the end users.  Spaces and Pages were introduced in S/4HANA 2020 which requires the configuration of Catalogs, Spaces, and Pages.  Prior to S/4HANA 2020 the Fiori Launchpad was configured using Catalogs and Groups.
There is also a consideration as to which method you are using currently (Catalogs and Groups or Catalogs and Spaces/Pages) which will determine the configuration steps you will follow below.  Please determine the method that you are currently using and follow the necessary steps.
Prerequisites:
Fiori Gateway Activated for the system
Workbench Transport created via transaction SE10
Custom Package created via transaction SE80 (recommend package: ZDTLS)
Necessary authorizations defined for Fiori Administration
End users already have the required Fiori Baseline access

### [Heading2] Custom Catalogs
To assign the different applications to the end users a custom Catalog needs to be created with the custom Tile configuration for each application.  Note that one or more Catalog’s may need to be created.  
The Catalog defines the applications that the end users can access in their Fiori Launchpad.  Segregating apps out into different roles requires different Catalogs as an end user will see all the apps that are included in the Catalog(s) assigned to their role.

### [Heading2] Authorization Object: ZDTLS_S
The main security auth object that controls access to DTLS applications is ZDTLS_S. This object will provide access to all applications. 
The ZDTLS_AUTH field allows the customization of access for all DTLS applications. 
The ACTVT field allows the restriction of activities within each DTLS application .
The ZDTLS_STGE field is currently not in use.
The possible values for field ACTVT are as follows. These are selected based on what each assigned role needs to be able to complete. Multiple activities can be selected to customize the access of any given role.
The possible values for auth field ZDTLS_AUTH are as follows. Each authorization value corresponds to a DTLS application.

### [Heading2] Workflow Functionality
If Workflow functionality is in scope, the following roles are required.

### [Heading3] Workflow Configuration Role
The purpose of the Workflow Admin role is to provide access to the DTLS Manage Process Workflow Rules app. This app is used to configure the workflow rules for all DTLS objects. 
Role Requirements:
Fiori Catalog assigned that contains the DTLS Manage Process Workflow Rules app
Fiori Space that contains the DTLS Manage Process Workflow Rules app
In the auth object ZDTLS_S, populate the ZDTLS_AUTH field with the ZDTLS_WF value and the ACTVT field with 01, 02, and 03
Please see below sections 3.4.2 and 3.4.3 as the roles defined in those sections are required for workflow configuration.

### [Heading3] Workflow Notification Role
This role is used in the Workflow Rule configuration and is used to determine the Users who will receive an email notification when a DTLS object is submitted for Approval.
The role does not require any specific authorizations to be added, it only needs to have the users assigned who should get the Notifications. 
The technical role name needs to be added during Rule Creation in the ‘DTLS Manage Process Workflow Rules’ application.
The technical name of the role needs to be provided to the resource that is performing the workflow configuration tasks. Also, this role is to be assigned to end users that should receive workflow notifications.

### [Heading3] Workflow Approver Role
This role is used in the Workflow Rule configuration and is used to determine the Users who will receive Workflow Approval requests when a DTLS object is submitted for Approval.  These users will receive an email notification and will be able to see their Workflow Tasks in the Fiori My Tasks application. See “DTLS Functional Guide: Workflow Approval” for more information on Workflow and the My Tasks application.1
The role does not require any specific authorizations to be added, it only needs to have the users assigned who will be required to approve submitted DTLS Object creations or changes.
The technical role name needs to be provided to the resource that is performing the workflow configuration tasks to be added during Rule Creation in the ‘DTLS Manage Process Workflow Rules’ application.
If digital signatures are being used, the following authorizations should be added to this role.
Auth Object: C_SIGN
ACTVT = 73
SIGNAPPL = ZDTLSOBJ
SIGNOBJ = ZDTLSOBJ

### [Heading1] Common Issues and Troubleshooting Steps

### [Heading3] 4.1 - PFCG Custom ZDTLS TADIR Service does not show up in PFCG
Custom DTLS TADIR services are required to be added using PFCG when initially setting up DTLS specific security roles. If there is an instance where one of these services does not show up in PFCG, the following instructions may be able to mitigate the issue:
Execute transaction SU25, it will show a series of utilities. Select the bottom most option titled 'SU22 Registration of Applications'.
Select the dropdown for “Type of Application”, then select “18. SAP Gateway: Service Groups Metadata”, untick the option “Exclude local applications”, and type *DTLS* in the object name field, as such:
*Note: when executing the aforementioned step, run it with “Display Mode for Affected Applications” checked.
Click the execute button. This will show all missing PFCG values that will be generated.
After confirming the values that are needed, go back one screen and execute Step #3 again, this time with the “Display Mode for Affected Applications” box unchecked.
Upon clicking execute once more with the display box unchecked, the utility will run and will register the missing services. There will be a small message indicating that the utility was successful:
Repeat steps 2-5 once more, this time selecting “19 - SAP Gateway Business Suite Enablement - Service” in the “Type of Application” dropdown
After executing these steps for both SAP Gateway: Service Groups Metadata and SAP Gateway Business Suite Enablement - Service, all missing authorizations should now be visible in PFCG for assignment to roles.
